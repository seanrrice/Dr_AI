# Doctor AI — Subsystem Output Specification

**Version:** v0.1
**Purpose:** Define how subsystems (Face, Audio, Gait) output data for the Doctor AI integration pipeline.

---

# 1. System Overview

Doctor AI integrates multiple sensing subsystems into a unified patient report.

Subsystems include:

* **Face** — facial emotion analysis
* **Audio** — speech transcription and semantic analysis
* **Gait** — motion and gait analysis

Each subsystem runs independently and writes its results into a shared **visit folder**.

A central **integrator** reads these outputs and generates the final report.

---

# 2. Integration Architecture

```
             Orchestrator
        (visit/session manager)
                 |
    --------------------------------
    |              |              |
   Face           Audio           Gait
  face.jsonl     audio.jsonl     gait.jsonl
    \              |              /
     \             |             /
      ----------- Integrator ----------
                     |
                 report.json
                     |
                    GUI
```

## Key Design Principle

Subsystems **do not communicate with each other directly**.

Each subsystem only needs to:

1. Receive visit context
2. Write its own JSONL output
3. Follow the shared data format

---

# 3. Visit Folder Structure

Each visit has a dedicated folder.

Example:

```
runs/
  visit_V001/
    manifest.json
    face.jsonl
    audio.jsonl
    gait.jsonl
    report.json
    figures/
```

## File Descriptions

| File            | Description                       |
| --------------- | --------------------------------- |
| `manifest.json` | Visit metadata and system state   |
| `face.jsonl`    | Face subsystem output             |
| `audio.jsonl`   | Audio subsystem output            |
| `gait.jsonl`    | Gait subsystem output             |
| `report.json`   | Integrated report                 |
| `figures/`      | Plots generated by the integrator |

Subsystem teams only need to produce their **own JSONL file**.

---

# 4. JSONL Format

Subsystems output **JSON Lines (JSONL)**:

* One JSON object per line
* Allows streaming and incremental writes

Example:

```
{record1}
{record2}
{record3}
```

---

# 5. Required Envelope Fields

Every JSONL record must contain the following fields:

| Field            | Description                                 |
| ---------------- | ------------------------------------------- |
| `visit_id`       | Unique visit identifier                     |
| `patient_id`     | Patient identifier                          |
| `subsystem`      | Name of subsystem (`face`, `audio`, `gait`) |
| `phase`          | Visit phase (`entry` or `encounter`)        |
| `type`           | Record type (`summary`, `window`, `event`)  |
| `features`       | Subsystem-specific metrics                  |
| `confidence`     | Confidence score                            |
| `valid`          | Boolean validity flag                       |
| `schema_version` | Output schema version                       |

---

# 6. Time Reference

All timestamps are measured relative to a shared time origin.

```
t0 = manifest.created_utc
```

Subsystem timestamps should represent:

> **seconds since visit start**

This allows data from different subsystems to align in time.

---

# 7. Record Types

## 7.1 Summary Records

Summary records represent aggregated information for a phase or entire visit.

Example:

```json
{
  "visit_id": "V001",
  "patient_id": "SeanTest",
  "subsystem": "face",
  "phase": "encounter",
  "type": "summary",
  "t_start": 2.1,
  "t_end": 15.8,
  "features": {
    "total_samples": 25,
    "emotion_counts": {
      "angry": 1,
      "disgust": 2,
      "happy": 15,
      "low_affect": 4,
      "arousal": 3
    },
    "emotion_pct": {
      "angry": 4.0,
      "disgust": 8.0,
      "happy": 60.0,
      "low_affect": 16.0,
      "arousal": 12.0
    },
    "pct_scale": "0-100"
  },
  "confidence": 1.0,
  "valid": true,
  "schema_version": "v0.1"
}
```

---

## 7.2 Window Records

Window records describe metrics computed over a time interval.

Example:

```json
{
  "visit_id": "V001",
  "patient_id": "SeanTest",
  "subsystem": "audio",
  "phase": "encounter",
  "type": "window",
  "t_start": 20.0,
  "t_end": 30.0,
  "features": {
    "word_count": 42,
    "diagnostic_term_pct": 0.08
  },
  "confidence": 0.84,
  "valid": true,
  "schema_version": "v0.1"
}
```

---

## 7.3 Event Records

Event records describe discrete events.

Example:

```json
{
  "visit_id": "V001",
  "patient_id": "SeanTest",
  "subsystem": "gait",
  "phase": "entry",
  "type": "event",
  "t": 18.2,
  "features": {
    "event": "walk_end"
  },
  "confidence": 0.95,
  "valid": true,
  "schema_version": "v0.1"
}
```

---

# 8. Phase Conventions

Two visit phases are currently defined.

## Entry Phase

Used for:

* gait / motion analysis
* patient entering the room

Subsystem:

```
gait
```

---

## Encounter Phase

Used for:

* face emotion analysis
* audio / speech analysis

Subsystems:

```
face
audio
```

---

# 9. Subsystem Responsibilities

Each subsystem team must:

1. Write `<subsystem>.jsonl` in the visit folder
2. Use the shared envelope structure
3. Provide at least one `summary` record

Optional improvements:

* add `window` records
* add `event` records
* include confidence metrics

---

# 10. Integrator Responsibilities

The integrator performs the following tasks:

1. Reads `manifest.json`
2. Reads subsystem JSONL files
3. Extracts summary statistics
4. Generates figures
5. Writes the final report

Output artifacts:

```
report.json
figures/*.png
```

---

# 11. GUI Responsibilities

The GUI reads **only the integrated report**.

The GUI should **not parse subsystem JSONL files directly**.

Input:

```
report.json
```

The GUI displays:

* subsystem summaries
* plots
* serial trend analysis
* synthesized biomarkers

# 13. Integration Rule of Thumb

Each subsystem should aim for the following **minimum output**:

```
1 JSONL file
1 summary record
correct envelope fields
```

Once this is working, additional windowed or event data can be added.

---

# 14. One-Sentence Integration Summary

Each subsystem writes its own JSONL file into the visit folder using the shared envelope structure, and the integrator combines these outputs into a unified report.
